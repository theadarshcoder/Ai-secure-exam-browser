01 — Problem Statement

Why does Vision exist?

Remote exams have a serious trust problem. Without physical supervision, students can cheat, impersonate others, or take exams in an uncontrolled environment — and nobody knows until it is too late.

😈 Cheating & Impersonation

Students use second devices, get answers from others, or have someone else take the exam on their behalf. There is no way to verify who is actually sitting in front of the screen.

👁️ No Real-Time Oversight

Mentors have zero visibility into what students are doing during the exam. Tab switching, window hopping, and looking up answers go completely undetected.

📋 Manual Grading Overload

Code answers and short answers require hours of manual review. For 100+ students, this becomes impossible without a smart queuing and grading system.

✅ What Vision Solves

Vision is a full-stack, production-grade exam platform that enforces identity verification before entry, locks the browser during the exam, captures real-time violations, enables mentors to monitor and intervene live, and grades code automatically — all in one system.

02 — Architecture

System Architecture

Vision follows a classic three-tier architecture — a React frontend, a Node.js/Express backend, and a data layer made up of MongoDB, Redis, and third-party cloud services.

Fig 1. Three-tier system architecture showing all components and their connections

🖥️ Client Layer

React 18 SPA with three separate portals — Student, Mentor, Admin. Uses Framer Motion for animations and Monaco Editor for the in-browser coding IDE.

⚙️ Application Layer

Node.js + Express handles all REST APIs and WebSocket connections. RBAC middleware, Bull job queues, and a Puppeteer scraper all run here.

☁️ Data & Services Layer

MongoDB stores all data. Redis handles caching and rate limiting. Cloudinary stores media. Judge0 runs student code in 40+ languages safely.

05 — Security

RBAC — Authorization Pipeline

Every request goes through a strict authorization pipeline. If the JWT is invalid or the user's role does not have permission for the route, the request is rejected before it ever touches the database.

Fig 4. Full request lifecycle — JWT verification, role check, rate limiting, and response

4 Roles in the System

  • student — Can only take assigned exams, submit answers, upload eKYC
  • mentor — Can create exams, import questions, grade answers, proctor
  • super_mentor — Elevated mentor with additional admin access
  • admin — Full platform access: users, eKYC, audit logs, RBAC assignment

Security Layers

  • JWT — All APIs require a valid token, signed with a secret key
  • Bcrypt — Passwords are hashed with 10 salt rounds, never stored plain
  • Rate Limiting — Redis blocks 100 req/15min general, 5 req/min for login
  • Helmet.js — HTTP headers sanitized against XSS and clickjacking
  • CORS — Only the frontend origin is whitelisted for API access
06 — Database

Database Schema

Vision uses MongoDB with 6 collections. The schema is designed for performance — indexed on the most common query patterns and split into separate collections where documents would otherwise exceed MongoDB's 16MB limit.

Fig 5. ER diagram for all 6 MongoDB collections with fields, types, indexes, and relationships

CollectionPurposeKey FieldsIndexes
users All user accounts role, isVerified, profilePicture, idCardUrl email (unique)
exams Exam definitions with embedded questions creator, status, questions[] (MCQ/Short/Coding) creator+createdAt, status
examsessions Each attempt by a student violations[], snapshots[], questionStates, remainingTimeSeconds exam+student (unique), status, student+status
examanswers Per-question student responses sessionId, answer, marksObtained, status sessionId+questionId (unique)
auditlogs Admin action history adminId, action, targetUserId, details createdAt
errorlogs Backend error tracking message, stack, route, userId createdAt

⚡ Why ExamAnswers is a separate collection

Storing all student answers inside ExamSession would cause documents to exceed MongoDB's 16MB per-document limit for large exams with coding questions. By splitting answers into their own collection with a sessionId foreign key, each document stays small and queries stay fast.

08 — Scaling

Scaling Discussion

Vision is designed with scaling in mind from day one. Here is how each layer handles growth — from 10 students to 10,000.

ChallengeCurrent SolutionAt Scale
High API Traffic Redis rate limiter prevents spam. Express handles concurrent requests. Add horizontal Node.js replicas behind a load balancer (Nginx). Redis stays shared.
Heavy Code Grading Bull Queue pushes grading jobs async so API never blocks. Add more Bull workers. Point to a self-hosted Judge0 cluster for higher throughput.
Real-Time WebSockets Socket.IO rooms group students by exam session. Use Redis Adapter for Socket.IO so multiple Node instances share the same socket state.
Media Storage Cloudinary handles all eKYC photos and snapshots — zero local disk usage. Cloudinary scales automatically. No changes needed on our side.
Database Growth MongoDB indexed on creator, status, student+status, sessionId+questionId. Enable MongoDB Atlas auto-scaling. Add read replicas for dashboard queries.
ExamSession Size Answers are split into ExamAnswers collection to avoid the 16MB document limit. Already solved by design. Works for exams with 500+ questions.

🏗️ What's already production-ready

  • Redis rate limiting protects against DDoS and brute force
  • Bull Queue isolates grading from the main API process
  • All DB queries use compound indexes — no full collection scans
  • ExamSession tracks remainingTimeSeconds server-side — timer can't be cheated client-side

🔜 Next scaling step

  • Add a Redis Adapter for multi-instance Socket.IO
  • Deploy multiple Node.js instances behind Nginx
  • Move to MongoDB Atlas for managed auto-scaling
  • Add a CDN layer (Cloudflare) in front of the React build
09 — Roadmap

Future Roadmap

Vision has a strong foundation. These are the features planned for the next phases of development.

🤖 AI Answer Evaluation Next

Use an LLM (GPT-4 / Gemini) to automatically evaluate short-answer and subjective responses. The AI grades phrasing and concept accuracy — reducing mentor workload by 80%.

🎭 Real-Time Face Recognition Next

Use a face detection model (face-api.js or AWS Rekognition) to match the live webcam feed against the eKYC selfie in real-time. Auto-flag if face is absent or does not match.

📊 Advanced Analytics Dashboard Planned

Per-exam performance heatmaps, question difficulty analysis, student risk scores, and violation trend graphs — giving mentors data-driven insight into exam quality.

📱 Mobile App (React Native) Planned

A dedicated mobile app for mentors to receive live proctoring alerts, approve eKYC requests, and view exam stats from anywhere.

🔗 Third-Party LMS Integration Planned

Connect Vision with Moodle, Canvas, or Google Classroom via REST APIs — so institutions can use Vision as the secure exam layer on top of existing systems.

🌐 Multi-Language Support (i18n) Planned

Add internationalization so the platform can be used in Hindi, Marathi, Tamil, and other regional languages — expanding the user base across India.
